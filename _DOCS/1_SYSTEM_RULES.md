## 🔧 Stack التقني الفعلي المطبق حالياً

- **Framework:** Next.js 14 (App Router)
- **Database (المرحلة الحالية):** Zustand State Management مع `persist` middleware لحفظ البيانات مؤقتاً في `localStorage`.
- **Database (الهدف المستقبلي):** Supabase (PostgreSQL + RLS) بمجرد النشر الإنتاجي (Production).
- **Language:** TypeScript (strict mode — منع استخدام `any` واستبداله بالأنواع الصحيحة كـ `UserPermissions`).
- **Styling:** Tailwind CSS فقط (تم تنظيف تعارضات CSS مثل flex/block).
- **Charts:** Recharts
- **Icons:** Lucide-React
- **State Management:** Zustand لمخازن (Auth, Data) مع دعم `JSONStorage`.

## 💰 العملة والأرقام (قانون لا يُكسر)

- **العملة الوحيدة:** دينار ليبي (LYD)
- **تنسيق الأرقام:** `1,500 د.ل` (بدون خانات عشرية — `Math.round` يقرّب لأقرب عدد صحيح)
- **التخزين المالي المطبق:** يتم استخدام دوال جاهزة `formatCurrency` للـ UI مع إجراء تقريب رقمي للحسابات المالية (بدون كسور غريبة).
- **القاعدة الذهبية:** نسبة الشركاء مجتمعة **لا يمكن** أن تتجاوز 100% مطلقاً (مراقبة برمجية نشطة في واجهة الشركاء تمنع السيرفر من الحفظ).
- لا يمكن إضافة ديون أو مبالغ بدون تحديد وجهتها.

## 🏗️ الهيكل المعماري الفعلي المطبق — طبقتان رئيسيتان

### 👑 الطبقة الصفرية: Super Admin (مدير منصة Bunyan)

إدارة شاملة لجميع المتاجر في الـ Local Storage:

- **دخول مخصص:** عبر `super@bunyan.ly`.
- **نظام التصفح الداخلي (Tenant Browsing):** يمكن لمدير المنصة دخول أي متجر كـ Owner عبر دالة `enterTenantAsAdmin` مع الحفاظ على هويته الأصلية للعودة ببساطة.
- إدارة المستأجرين (التجار): إضافة متاجر، تفعيل وتجميد (Kill-Switch). المُتجر الموقوف يُطرد مستخدموه فوراً في الـ Layout.
- إحصائيات عامة عبر كافة المتاجر.

### 🏢 الطبقة الأولى: بيئة التاجر (Tenant Workspace)

كل تاجر يملك `tenantId` خاص به يعزل بياناته.

- يتم فلترة كل الجداول بالاعتماد على دالة `getForTenant` ضمن الـ Zustand Store.
- إمكانية التجوال بأدوار متعددة عبر `setViewingAs` لفحص النظام بعين الموظف أو الشريك.

## 🔐 نظام Auth والمصادقة الحالي

بانتظار Supabase، تم تطبيق نظام مصادقة آمن محلياً للـ Demo:

- **localStorage + Cookie:** يتم حفظ الجلسة في `erp_auth` كـ Cookie ليمنع الـ Middleware وصول غير المصرح لهم، ويتم قراءة تفاصيل المستخدم من Zustand `persist`.
- **التشفير المؤقت:** الباسوردات مشفرة برمجياً بـ `btoa()` وتُقارن به.
- **إنشاء مستخدمين صامت (Silent User Creation):**
  - عند إنشاء موظف جديد في الـ HR → يتولد له حساب `TenantUser` بصلاحية `employee` + Password افتراضي (123456).
  - عند إنشاء شريك جديد → يتولد له حساب `TenantUser` بصلاحية `partner`.

## 👑 هرم الصلاحيات المطبّق (UserPermissions)

كل حساب `TenantUser` يمتلك كائن `permissions` مخصص يُقيّد صلاحياته:

```json
{
  "inventory": {
    "view": true,
    "add": false,
    "edit": false,
    "delete": false,
    "viewCostPrice": false
  },
  "orders": {
    "view": true,
    "add": true,
    "edit": false,
    "delete": false,
    "changeStatus": true,
    "viewAll": false
  },
  "delivery": {
    "view": true,
    "addShipment": true,
    "manageCompanies": false,
    "viewSettlements": false,
    "addSettlement": false
  },
  "treasury": { "view": false, "addTransaction": false },
  "partners": { "view": false, "viewOwn": true },
  "hr": { "view": false, "viewOwn": true },
  "analytics": { "view": true, "viewFull": false },
  "settings": { "view": false, "edit": false }
}
```

- **OWNER**: صلاحيات كاملة `True` في كل شيء.
- **EMPLOYEE**: البيع والتوصيل ولا يرى الخزائن ولا أسعار التكلفة.
- **PARTNER**: يرى أرباحه وديونه والإحصائيات الخاصة به فقط ويعجز عن دخول الإعدادات أو الموارد البشرية.

## ⚙️ وحدة الإعدادات المتقدمة (Settings)

تم نقل كافة إعدادات النظام لصفحة `settings` مقسمة إلى 6 تبويبات فعّالة:

1. **عام (General):** التحكم باسم المتجر، الشعار (هوية بصرية)، الرقم الضريبي.
2. **الفوترة (Billing):** اشتراك المنظومة (مدفوع/تجريبي)، التجديد.
3. **الطلبيات (Orders):** الحقول الإلزامية وتخصيص البادئات.
4. **المخزون (Inventory):** تنبيهات نفاد المخزون ووحدات القياس.
5. **الإشعارات (Notifications):** أصوات التنبيه والخيارات الزمنية.
6. **المستخدمون (Users):** إضافة / دعوة مستخدم جديد وتحديد دوره والصلاحيات (Permissions).
7. **الأمان (Security):** تغيير كلمة المرور والتصدير وتصفير الداتا.

## 🚫 محظورات مطلقة (عبر الكود الحالي)

- ❌ لا `any` في Typescript — تم إصلاح كل المتغيرات لـ `TenantUser` و `UserPermissions` وإزالة `as any` من Store وAddProductSlideOver.
- ❌ لا حسابات دائرية (Infinite Redirects) بين Zustand والـ Middleware — تم إصلاح المتلازمة عبر ضبط المزامنة بـ `setTimeout`.
- ❌ لا يتم حذف ملفات CSS بشكل عشوائي، وتمت معالجة تحذيرات `block` و `flex` المزدوجة التي تضر بالـ DOM.
- ❌ نسبة الشركاء لا تتخطى 100% (منطق معالَج في كود الإضافة / التعديل للشركاء).
- ❌ لا يُسمح بتجاوز الأدوار برمجياً: الـ UI مؤمن، والـ Middleware يحمي `/settings` و `/treasury` و `/hr` و `/analytics` و `/super-admin` حسب الدور.
- ❌ تجنب استخدام `sku` لتعريف المنتجات: تم اعتماد رمز تلقائي تسلسلي (`itemCode` يبدأ من 1000 ويعرض `BN1000`) لتحاشي تكرار الرموز وأخطاء الإدخال.
- ❌ لا يُخصم المخزون من الرصيد العام في حال كان المنتج يحتوي على متغيرات (Variants)، الخصم يحترم أعداد المتغيرات بدقة لكل الأنواع (`clothing`, `shoes`, `custom`).
- ❌ يمنع إضافة منتجات جديدة أو تعزيز كميات دون خصم قيمتها المالية من الخزينة تلقائياً. المزامنة المالية الفورية للمخزون إجبارية.
- ❌ لا يُسمح بخصم تكلفة البضاعة بسعر ثابت — يجب استخدام متوسط التكلفة المرجح (WAC) دائماً عند إضافة كميات جديدة.
- ❌ لا يُسمح بإضافة كميات أو منتجات إذا كانت التكلفة الإجمالية تتجاوز رصيد الخزينة (cash_in_hand) — يجب تعطيل زر الحفظ وإظهار رسالة خطأ.
- ❌ الحماية من الخزينة السالبة يجب أن تكون في Store Layer وليس UI فقط — addProduct و updateProduct يجب أن يتحققا من الرصيد قبل أي خصم.
- ❌ ممنوع استخدام .toFixed(2) للأرقام المالية — استخدم Math.round فقط.
- ❌ customCategories و customUnits يجب عزلها بـ tenantId — ممنوع تخزينها كمصفوفة عامة.
- ❌ شريط تصفية الفئات في المخزون يجب أن يكون ديناميكياً — ممنوع استخدام فئات ثابتة (Hardcoded).
- ❌ عند حذف موظف أو شريك → يتم تعطيل حسابه `TenantUser` المرتبط تلقائياً.
- ❌ ممنوع كتابة كود API لأي شركة توصيل داخل store.ts أو أي page — كل شركة لها ملف Adapter مستقل في src/core/delivery/
- ❌ ممنوع استدعاء fetch() أو axios لشركات التوصيل من خارج ملف الـ Adapter الخاص بها
- ❌ ممنوع إضافة شركة توصيل جديدة بدون إنشاء ملف [CompanyName]Adapter.ts يُطبّق واجهة IDeliveryProvider
